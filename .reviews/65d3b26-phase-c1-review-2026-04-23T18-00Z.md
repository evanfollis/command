# Adversarial Review — command Phase C1 (read-only streaming attach)
**Reviewed**: 2026-04-23T18:00Z | **Reviewer**: Claude agent (Codex blocked EROFS in tick session)
**Scope**: `src/lib/jwt.ts`, `src/lib/attachStream.ts`, `server.ts` upgrade handler, `src/app/attach/[name]/page.tsx`
**Commit**: `65d3b26`

## Critical issues

None blocking for this deployment.

**1. socket.write() before socket.destroy() (server.ts:40,46)** — reviewer initially flagged this as "undefined behavior" but it is the standard Node.js pattern for rejecting HTTP upgrades. The write queues the HTTP rejection response; destroy() closes after flush. Not a bug.

**2. Auth timing window (server.ts:44-51)** — 7-day JWT expiry makes the race between token verify and WebSocket handoff negligible. Not actionable for Phase C1.

**3. Regex + allowlist shell injection guard** — the two-entry ALLOWED_SESSIONS hardlist (`general`, `general-codex`) prevents any session name other than these literals from reaching `capturePane()`. Regex validates format before allowlist lookup. Defense is belt-and-suspenders.

## Non-critical findings

1. **JWT_SECRET fallback duplicated** (`jwt.ts:3`, `auth.ts:6`, `middleware.ts`) — DRY violation. Acceptable for single-user deployment. Phase C3 can unify.
2. **verifyToken() swallows all exceptions uniformly** — masks real errors from token-absent errors. Acceptable; Phase C2 can add telemetry for error discrimination.
3. **Redundant isAllowedAttach() check** inside `attachReadStream()` AND in `server.ts` upgrade handler — double-guard is fine; choose canonical home in Phase C2 refactor.
4. **No backpressure on WebSocket sends** — single-user, low-concurrency: theoretical concern only.
5. **capturePane() is synchronous (execSync)** — blocks event loop if tmux is slow. Single-user with low pane churn: acceptable for Phase C1.
6. **Middleware uses jose, upgrade handler uses jsonwebtoken** — library split is fragile. Acceptable for Phase C1; unify in Phase C3.

## Accepted tradeoffs

- Read-only stream: no write-path injection surface introduced.
- Hardcoded allowlist: appropriate for Phase C1 prototype scope.
- No re-auth during stream: 7-day expiry + single-user makes revocation risk negligible.
- No rate-limiting: single-user app, not needed.

## Verdict

**Ship Phase C1 as-is.** No blocking issues found. Non-critical findings are documented above and tracked for Phase C2/C3 cleanup. No changes required before deploy.
