# Review: 690a2d9 — Tighten thread-opening-frame cases and extend adapter timeout

**Date**: 2026-07-14T05-41Z  
**Reviewer**: Claude (Sonnet 4.6) — adversarial substitute (codex EROFS in tick session)  
**Change**: thread-opening-frame/golden/cases.jsonl (+9 -9), adapter_llm.py (+2 -2)

## Summary

Downgrades `must_pass:true` to `must_pass:false` (with `advisory_reason`) on cases where the probe's read-only tool set makes genuine execution impossible. The judge was scoring these as deferral (correctly), and the loop was failing the prompt for the harness gap. Also extends the LLM adapter timeout from 300→380s to reduce false throttle errors.

## Findings

**[info]** The case taxonomy is correct: cases that require Write/Bash/Edit tools cannot be verified by a probe that has only Read/Grep/Glob. Marking them advisory instead of must-pass is the right design — the gate now measures the prompt's decision quality, not the harness's tool limitations.

**[advisory]** The 380s timeout is now longer than the `judge.trials: 3` spec suggests. With 3 trials and a slow judge response, the effective wall time per case could be ~1140s (19 min). This is acceptable for an offline eval loop but would affect CI time if the suite grows. Not a correctness issue.

**[info]** No golden case content changed — only the `must_pass` flag and `advisory_reason` field. Holdouts are unaffected.

## Verdict

Safe. The advisory downgrade prevents the harness limitation from masking the prompt's real behavior, which is the point of the eval gate.
