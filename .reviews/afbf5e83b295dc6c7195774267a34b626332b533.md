# Review: afbf5e8 — Serve from immutable releases so a build cannot corrupt a running process

**Date**: 2026-07-14T05-41Z  
**Reviewer**: Claude (Sonnet 4.6) — adversarial substitute (codex EROFS in tick session)  
**Change**: release.sh (+120), rollback.sh (+12), smoke.ts (+19), package.json (+5)

## Summary

Eliminates the split-brain failure class where `next build` in the live working directory could silently replace `.next/` assets while the running process served them. The service now runs from an immutable release tree at `runtime/releases/command/current`; a build in the repo can only touch the repo's own `.next`, which nothing serves. Atomic `rename(2)` swap; smoke-gated rollback.

## Findings

**[info]** `release.sh` correctly uses a throwaway git worktree (not the live tree), marks the release read-only after assembly, and uses `rename(2)` for the atomic cutover. The reasoning in the commit message is sound.

**[info]** `smoke.ts` now asserts every asset referenced by authenticated HTML returns 200. This closes the gap where the prior health endpoint and `/login` CSS check both missed the outage — the outage only affected authenticated route chunks.

**[info]** `dist/.version` records true SHA and dirty state. A dirty release reports as `<sha>-dirty` via `/api/health` — no silent false-clean.

**[info]** The rollback script re-points `current` and verifies service active + `/login=200` before declaring success. Correct.

**[advisory]** `ALLOW_DIRTY=1` path runs `scripts/assert-dirty-release-inputs.sh` but the script exists only in a later commit (c28fd43). At the time of this commit, `ALLOW_DIRTY=1` would silently pass if the script was absent. Not a concern given the linear commit ordering and the script's presence in HEAD.

## Verdict

Safe. This commit closes a real failure class with sound architectural choices. The atomic swap, immutable tree, and authenticated smoke coverage are all correct. Advisory item is a historical ordering note, not a live concern.
