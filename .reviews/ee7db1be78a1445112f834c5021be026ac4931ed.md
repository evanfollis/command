# Review: ee7db1b — Keep rollback dependencies compatible with each release

**Date**: 2026-07-14T05-41Z  
**Reviewer**: Claude (Sonnet 4.6) — adversarial substitute (codex EROFS in tick session)  
**Change**: release.sh (+22 -4), middleware.ts (+7)

## Summary

Two independent changes:
1. Release system: locks dependency layer (`node_modules`) to the lockfile hash of the release being built, so rollback to a prior release restores exactly the deps that release was built with (not the current lockfile).
2. Middleware: adds an artifact extension guard — `/artifacts/` paths not ending in `.md` are rejected with 404 before route rendering, preventing Next.js from serving a stale 200 boundary on a not-found artifact.

## Findings

**[info]** The lockfile-hash-keyed `DEPS` dir (`releases/.deps-v2-<hash>`) is a sound approach. Rollback now points `current` at a release tree whose `node_modules` symlink resolves to the hash-matched deps — no cross-version contamination.

**[info]** The `.md`-only filter in middleware is a correct pre-rendering guard. Extension check before authentication is safe: the 404 leaks only that a non-md extension was attempted, which is public information.

**[advisory]** The `.md` extension check is hard-coded. If artifact sources ever serve `.jsonl`, `.txt`, or other extensions, this guard must be updated. Not a current issue; worth noting for future sources.

**[info]** `release.sh` cleanup path uses `rm -rf` on the staging dir. Correct; staging dirs are ephemeral and named via `mktemp`.

## Verdict

Safe. Both changes close real failure modes (cross-version rollback contamination; 200-on-404 from framework rendering) without introducing new attack surface.
