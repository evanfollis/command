## Summary

Adds `validateProjectionV1()`: an exact-key schema validator that rejects any public projection JSON that doesn't match the typed v1 contract shape (exact field names, types, nested structure). Replaces the loose `recordCounts` accumulator with typed `counts` + `provenance` fields. Adds `publicCoherence` and `ownerQueueState` signals. The validator fails closed — any schema deviation returns `availability: 'invalid'`, not a partial read.

## Findings

**[info]** `hasExactKeys()` verifies both required keys are present AND no unexpected keys exist (bidirectional check). This is correct strict validation — it prevents silent schema extension from leaking extra fields through.

**[info]** `validateProjectionV1` throws on the first violation. The caller (`collectPublicProjection`) wraps this in `try/catch` and returns `availability: 'invalid'` on throw. Fail-closed behavior is correct.

**[advisory]** The `research[*].provenance` validator checks `claim_id`, `decision_id`, `evidence_ids` exactly. If the v1 contract adds an optional field (e.g. `notes`), all existing projections would fail validation until the validator is updated. Strict validation is appropriate for this domain (cryptographic projection integrity), but the team should be aware that schema evolution requires a co-update here.

**[info]** The `ObservatoryDashboard.tsx` change (+4 -4) is a prop name rename following the `recordCounts` → `counts` refactor. No functional change.

**[info]** `release-invariants-test.sh` and `release-lib.sh` additions test the release system's own invariants. Appropriate.

## Verdict

Safe. The typed projection validator is sound and correctly fail-closed. The strict schema check is the right tradeoff for a system that derives signals from a versioned contract.
